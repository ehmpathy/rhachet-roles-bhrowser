# vision: inject prior browser skills

## the outcome world

### before / after contrast

| before | after |
|--------|-------|
| browser automation knowledge scattered in declastruct-squarespace | consolidated, reusable playwright role in bhrowser |
| no explicit teaching of caching, auth, stealth patterns | clear briefs teach each concept distinctly |
| ad-hoc browser skill implementations | dispatcher pattern routes to browser.* skills consistently |
| captcha handling unclear | tiered escalation strategy documented and reusable |
| session persistence implementation-specific | cross-session auth pattern teachable and portable |

### day-in-the-life

a mechanic building browser automation for a new site:

1. enrolls `repo=bhrowser/role=playwright`
2. reads briefs on stealth, caching, auth — understands the patterns before writing code
3. uses `browser.start` to launch persistent browser with stealth configured
4. uses `browser.action` to run playbooks against the site
5. uses `browser.snapshot` to capture diagnostic state when issues arise
6. when captcha appears, knows to escalate: headful pivot → human solve → session handoff
7. uses `withRemoteStateCache` pattern to avoid redundant page loads
8. tests pass, session persists across runs, no re-auth needed

### the "aha" moment

"oh — i don't have to figure out stealth, cache, and auth from scratch. the briefs teach the patterns, the skills implement them, and i can focus on my actual automation logic."

## user experience

### usecases

| usecase | goal | skills/briefs involved |
|---------|------|------------------------|
| launch browser | start persistent, stealth-configured chromium | `browser.start` |
| run automation | execute playwright playbooks | `browser.action` |
| diagnose issues | capture screenshots, html, console, storage, network | `browser.snapshot.*` |
| list tabs | see open pages for tab selection | `browser.describe` |
| stop browser | gracefully terminate session | `browser.stop` |
| understand stealth | learn detection signals and mitigations | stealth briefs |
| handle captchas | tiered escalation strategy | antibot briefs |
| persist auth | storageState across sessions | auth briefs |
| cache remote state | avoid redundant queries | cache briefs |
| diagnose failures | snapshot before assume, grep HTML for selectors | diagnosis briefs |
| verify actions | confirm actions succeeded, not just "didn't fail" | verification briefs |
| prevent hangs | bounded timeouts with fail-fast, bisection workflow | timeout briefs |

### contract inputs & outputs

**browser.start**
```
input:  --session <name> [--headless]
output: .cache/browser.$SESSION/ws-endpoint (CDP endpoint for reconnect)
```

**browser.action**
```
input:  --playbook <path.ts> [--session <name>] [--tab <index>]
output: playbook execution result, side effects on page
```

**browser.snapshot**
```
input:  --session <name> [--tab <index>] [--what screen|html|console|storage|network|meta|all]
output: .cache/browser.$SESSION/snapshot.{timestamp}/ with requested artifacts
```

**browser.describe**
```
input:  --session <name>
output: list of tabs with index, url, title
```

**browser.stop**
```
input:  --session <name>
output: browser process terminated, endpoint file cleaned
```

### timeline

1. **session start** — `browser.start` launches chromium, writes ws-endpoint
2. **auth (if needed)** — playbook logs in, writes storageState
3. **work** — `browser.action` runs playbooks, caches remote state
4. **diagnostics** — `browser.snapshot` captures state when needed
5. **session end** — `browser.stop` terminates (or leave running for reuse)
6. **next session** — reconnects via ws-endpoint, restores storageState

## mental model

### how users describe it

> "browser skills let me drive playwright from the shell. start a browser, run playbooks, take snapshots. it handles stealth and auth persistence so i don't have to."

### analogies

| analogy | mapping |
|---------|---------|
| browser as daemon | chromium runs persistently, skills connect via CDP |
| playbook as procedure | typescript file with playwright automation |
| snapshot as debugger | captures browser state for post-mortem |
| storageState as session cookie jar | persists auth across respawns |
| stealth as disguise | masks automation signals from detection |

### terms: theirs vs ours

| user term | our term |
|-----------|----------|
| browser | persistent chromium instance |
| playbook | typescript file with playwright api usage |
| snapshot | multi-format diagnostic capture |
| session | named browser instance (.cache/browser.$SESSION/) |
| stealth | playwright-extra + StealthPlugin |
| captcha escalation | tiered antibot strategy |

### debug movie frames

when tests fail, the snapshot workflow captures "movie frames" — screenshots and HTML captures that preserve browser state at key moments:

1. **test runs** — browser actions proceed, snapshots captured on demand via `browser.snapshot`
2. **failure occurs** — manual snapshot captures screen + HTML + console + network
3. **diagnosis** — grep HTML for `data-testid`, view screenshot, inspect console
4. **fix** — root cause visible in captured state, not guessed from error messages

note: snapshots are manual (on demand), not automatic after each action. this keeps overhead low while diagnosis remains possible when needed.

this "movie frame" approach saves 10-30 minutes per diagnosis session — actual page state reveals root cause, not assumptions.

### pros

- **reusable** — patterns work for any site, not squarespace-specific
- **teachable** — briefs explain the "why" not just the "how"
- **dispatcher pattern** — consistent with rhachet-roles-ehmpathy conventions
- **session persistence** — browsers survive skill exit, auth survives browser respawn
- **tiered escalation** — graceful degradation from stealth → headful → human

### cons

- **chromium-only** — skills assume chromium, not firefox/webkit
- **CDP dependency** — relies on chrome devtools protocol for reconnection
- **stealth arms race** — detection techniques evolve, stealth patches lag

### edge cases & pit of success

| edge case | how contracts keep users safe |
|-----------|------------------------------|
| no browser running | skills fail fast with clear error |
| wrong session name | skills fail fast, list available sessions |
| captcha detected | briefs teach escalation, not silent failure |
| storageState expired | auth playbook re-runs, persists new state |
| tab index out of range | skills fail fast, suggest `browser.describe` |

**withRemoteStateCache pattern (detailed breakdown):**

the pattern uses `with-remote-state-cache` npm package (ehmpathy/with-remote-state-cache) for automatic cache invalidation.

core concepts:
- `createRemoteStateCacheContext({ cache })` — creates context with cache backend
- `withRemoteStateQueryCache(fn)` — wraps queries to cache results
- `withRemoteStateMutationRegistration(fn)` — registers mutations for trigger system
- `addTrigger({ invalidatedBy: { mutation, affects } })` — connects mutations to queries

how declastruct-squarespace implements it:

```typescript
// 1. create context with SimpleOnDiskCache
const remoteStateCacheContext = createRemoteStateCacheContext<
  [any, ContextSquarespaceAgent],
  SimpleOnDiskCache
>({
  cache: ({ fromInput }) => fromInput[1].agentOptions.remoteState.cache,
  serialize: {
    key: (input, context) => [
      context.agentOptions.account.id,  // namespace to account
      JSON.stringify(input).slice(0, 100),  // preview for observability
      asHashSha256(JSON.stringify(input)),  // unique hash
      'v1',  // version for schema changes
    ].join('.'),
    value: (output) => serialize(output, { lossless: true }),
  },
  deserialize: {
    value: (cached) => deserialize(cached, { with: [DomainObject1, DomainObject2] }),
  },
});

// 2. wrap get operations
const { execute: getAllDomainsWithCache, addTrigger: addTriggerToGetAllDomains } =
  withRemoteStateQueryCache(
    withNewLoggedInBrowserPage(getAllDomainsFromPage),
    { name: 'getAllDomains' }
  );

// 3. mutations can trigger invalidation via addTrigger
```

why this matters for browser automation:
- browser page scrapes are expensive (seconds per page)
- cache avoids redundant scrapes when data unchanged
- mutations (set operations) automatically invalidate affected queries
- triggers fire even when mutations throw — cache never goes stale

compatible cache backends:
- `simple-on-disk-cache` — node.js filesystem (used in declastruct-squarespace)
- `simple-localstorage-cache` — browser localstorage
- `simple-dynamodb-cache` — aws dynamodb
- `simple-s3-cache` — aws s3

**snapshot/movie frames pattern (detailed breakdown):**

the pattern uses modular snapshot skills to capture "movie frames" — browser state at key moments for post-mortem diagnosis.

skill architecture:
- `browser.snapshot.sh` — dispatcher that runs all or single capture
- `browser.snapshot.screen.sh` — captures screenshot (PNG)
- `browser.snapshot.html.sh` — captures full HTML
- `browser.snapshot.console.sh` — captures console logs (JSON)
- `browser.snapshot.network.sh` — captures network requests (JSON)
- `browser.snapshot.storage.sh` — captures localStorage/sessionStorage (JSON)
- `browser.snapshot.meta.sh` — captures tab metadata (URL, title, viewport)

output structure:
```
.cache/browser.$session/snapshot.$isotime.tab$tab/
├── snapshot.meta.json
├── snapshot.png
├── snapshot.html
├── snapshot.console.json
├── snapshot.network.json
└── snapshot.storage.json
```

pit-of-success design:
- requires `--focused` OR (`--tab` AND `--url`) to prevent wrong-tab mistakes
- `--url` is verification key, not search — asserts tab N has this URL
- fail-fast if tab URL doesn't match expected

usage pattern:
```sh
# snapshot focused tab (simplest)
rhx browser.snapshot --focused

# snapshot specific tab with URL verification
rhx browser.snapshot --tab -1 --url 'account.squarespace.com/domains'

# capture just screenshot
rhx browser.snapshot screen --focused
```

diagnosis workflow (from briefs):
1. interaction fails → snapshot immediately (before hypothesis)
2. grep HTML for relevant elements (`dialog`, `modal`, `data-testid`)
3. form hypothesis based on actual state, not error message
4. fix based on evidence

time savings: 10-30 min per debug session → 2-5 min with snapshot-first approach.

**auth persistence pattern (detailed breakdown):**

the pattern uses `genBrowserAuthSession.ts` to create a mutable session handle with respawn capability for mode switch (headless ↔ headful).

key interface:
```typescript
interface BrowserAuthSession {
  browser: Browser;
  context: BrowserContext;
  mode: BrowserMode;  // 'HEADLESS' | 'HEADFUL'
  close: () => Promise<void>;
  respawn: (input: { mode: BrowserMode }) => Promise<void>;
  storage: {
    get: () => Promise<void>;  // restore storageState from disk
    set: () => Promise<void>;  // persist storageState to disk
  };
}
```

workflow:
1. `genBrowserAuthSession(opts)` — launch or connect to browser with stealth
2. check session health via test page load
3. if invalid → perform login → persist storageState
4. return mutable handle for all consumers

respawn capability:
- `handle.respawn({ mode: 'HEADFUL' })` — save session, close browser, relaunch in new mode, restore session
- enables captcha escalation: headless fails → respawn headful → human solves → respawn headless
- refuses mode change when connected to skill-started browser (use `browser.start --mode X --refresh`)

session persistence:
- `storage.set()` — write storageState JSON to disk (cookies + localStorage)
- `storage.get()` — re-create context with fresh storageState from disk
- path configured via `agentOptions.session.storageStatePath`

CDP connection:
- if `wsEndpoint` provided → `chromium.connectOverCDP(wsEndpoint)` to connect to skill-started browser
- reuses extant context from skill-opened browser (same window)

**note:** this TypeScript pattern is documented via briefs with exact code examples and references to `ehmpathy/declastruct-squarespace`. users implement their own version per the brief. SDK export may come later.

**antibot escalation pattern (detailed breakdown):**

3-tier approach to handle bot detection and captcha challenges:

| tier | trigger | strategy | human? |
|------|---------|----------|--------|
| 1 | prevention | stealth plugin masks automation signals | no |
| 2 | checkbox | headful pivot + robot click | no |
| 3 | puzzle | headful pivot + human solve | yes |

tier 1 (stealth):
```typescript
import { chromium } from 'playwright-extra';
import StealthPlugin from 'puppeteer-extra-plugin-stealth';

chromium.use(StealthPlugin());
const browser = await chromium.launch({ headless: true });
```

patches applied: `navigator.webdriver = false`, removes `HeadlessChrome` from UA, spoofs WebGL/plugins/fonts.

tier 2 (headful checkbox):
- detect captcha: `page.locator('iframe[src*="challenges.cloudflare.com"]').count() > 0`
- save storageState → close browser → relaunch headful
- robot clicks checkbox in iframe
- save storageState → resume headless

tier 3 (human puzzle):
- console prompt: "captcha puzzle detected — solve in browser window"
- `waitForEnter()` after human solves
- save storageState → resume headless

detection signals (why headful required):

| signal | headless | headful | stealth fix? |
|--------|----------|---------|--------------|
| `navigator.webdriver` | true | false | yes |
| user agent | HeadlessChrome | Chrome | yes |
| window focus | fake events | real events | no |
| canvas render | software | GPU | no |
| frame rate | no display | 60fps | no |

**browser.action playbook pattern (detailed breakdown):**

the pattern uses TypeScript playbooks for reusable browser automation sequences.

playbook format (`.play.ts`):
```typescript
import type { Page, Browser } from 'playwright';

export const action = async (input: { page: Page; browser: Browser }) => {
  await input.page.goto('https://example.com');
  await input.page.click('button');
  return { success: true };
};
```

directory structure:
- `.play/temporary/` — scratch work (gitignored)
- `.play/permanent/` — reusable sequences (committed)

usage:
```sh
rhx browser.action --play .play/permanent/goto-domains-list.play.ts
rhx browser.action --play .play/temporary/toggle-renewal.play.ts --tab 2
```

execution:
- connects to active browser via CDP (`chromium.connectOverCDP(wsEndpoint)`)
- gets page by tab index (default: 0)
- runs exported `action()` function with page and browser
- logs URL before/after and result

usecases:
- explore UI to discover selectors or URLs
- toggle settings or click buttons manually
- verify UI state (enabled vs disabled)
- perform one-off operations without full scrapers
- figure things out before production code

## what is awkward?

### feels off

1. **chromium direct launch vs playwright launch** — declastruct-squarespace launches chromium directly to persist across skill exit. this bypasses playwright's browser management. acceptable tradeoff for session persistence, but feels like fighting the tool.

2. **stealth as bolt-on** — playwright-extra wraps playwright, StealthPlugin patches webdriver signals. layered workarounds, not first-class support. works until it doesn't.

3. **tiered escalation requires human** — tier 3 (puzzle captcha) needs a human at a display. cannot fully automate. this is reality, not a design flaw, but feels incomplete.

### tradeoffs that feel uncomfortable

1. **session files in .cache/** — filesystem state for sessions. simple, but not portable across machines. acceptable for local dev, not for CI.

2. **CDP reconnection** — connecting to running browser via websocket endpoint. fragile if browser crashes. no automatic recovery.

3. **stealth patches lag detection** — cloudflare and others evolve. stealth plugin patches are reactive. users may need to update frequently.

### where design fights mental model

1. **"browser" is actually "chromium process"** — users might expect playwright browser object, but we provide CDP endpoint. different abstraction level.

2. **"session" is directory, not object** — session state is files in `.cache/browser.$SESSION/`. not an in-memory object. users must understand filesystem as state store.

3. **playbooks are files, not functions** — playbooks are `.ts` files executed via `npx tsx`. not imported functions. different invocation model than typical playwright procedures.
