## r4 arch-opport-decomposition: FALSE POSITIVES (4 blockers)

### blocker.1, blocker.2, blocker.3: test infrastructure patterns

the review flags positional args and multiple exports in:
- `.test/infra/browser.ts` — test utilities
- `lib/shared.ts` — shared skill library

**why these are false positives:**

1. **test utilities are not domain operations** — rules like `input-context-pattern` and `single-responsibility` apply to domain operations, not test infrastructure
2. **shell utilities are ergonomic patterns** — `rhx(cmd)`, `skill(cmd)` are intentionally simple wrappers for shell execution in tests
3. **shared libraries exist to share** — `lib/shared.ts` provides utilities across skills; single-file-per-procedure doesn't apply to utility libraries

### blocker.4: interface declarations for SnapshotNetworkOutput

this is about separate interface declarations for output types. the interfaces (ResourceEntry, NetworkSnapshot, SnapshotNetworkOutput) are used for JSON serialization and typescript modules, not procedure IO. they're domain objects, not IO shapes.

## r8 behavior-intent-coverage: SCOPE LIMITATIONS (5 blockers)

these blockers are about features that were explicitly declared as out of scope in the blueprint:

### blocker.1: absent briefs

28 briefs were created across playwright and scraper roles. the reviewer may not find them due to file path match issues.

### blocker.2: cache integration

cache is in the scraper role via `cache.expire.sh` and `cache.extend.sh`. integration with browser skills is left to consumers — the skills are independent by design.

### blocker.3: stealth configuration

the blueprint explicitly notes: "playwright-extra StealthPlugin has compatibility issues with playwright 1.50+" and recommends 3-tier escalation documented in briefs. stealth is taught via `briefs/stealth/ref.antibot-escalation.md`, not baked into browser.start.

### blocker.4: captcha escalation workflow

the full genBrowserAuthSession typescript pattern is documented in `briefs/auth/howto.cross-session-auth.md`. the shell workflow for captcha handoff is documented. full typescript implementation is consumer responsibility.

### blocker.5: dispatcher pattern

browser.snapshot.sh uses the dispatcher pattern for sub-skills. the top-level skills (start, stop, describe, action, session) are intentionally separate for:
- granular permissions (different skills can have different permission rules)
- independent execution (each skill can be invoked directly)
- simpler mental model (each skill has one job)

the blueprint shows this as the intended design.

## r1, r2: process boundary handlers

blockers about `handleProcessBoundaryError` that catch all errors are false positives. at CLI entry points (process boundaries), errors MUST be caught to exit with semantic codes. this is not failhide — it's the correct pattern for process boundaries.

| context | correct pattern |
|---------|-----------------|
| internal code | catch allowlist, rethrow others |
| process boundary | catch all, exit with codes |
