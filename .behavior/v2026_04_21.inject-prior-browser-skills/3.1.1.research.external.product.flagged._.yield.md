## topic.3 — CDP websocket endpoint reconnection stability

### summary

CDP connections via `connectOverCDP` are "significantly lower fidelity" than playwright protocol connections per official docs. known issues: timeout at protocol init (chrome m136+), connection decay after 5-6 hours idle, long-run chrome (12+ hours) degradation. no built-in automatic reconnection — requires manual implementation with keepalive pings.

### sources

#### [1] playwright documentation — browsertype api
**url:** https://playwright.dev/docs/api/class-browsertype
**type:** [official]
**date:** current

> "Connect over the Chrome DevTools Protocol is only supported for Chromium-based browsers."
> "This connection is significantly lower fidelity than the Playwright protocol connection via browserType.connect(). If you are in trouble or attempt advanced functionality, you probably want to use browserType.connect()."

- [FACT] `connectOverCDP` accepts `timeout` parameter (default: 30000ms, 0 to disable)
- [FACT] CDP connections "significantly lower fidelity" than playwright protocol
- [FACT] only chromium-based browsers support CDP connections

#### [2] github issue #35928 — connect_over_cdp stuck
**url:** https://github.com/microsoft/playwright/issues/35928
**type:** [official]
**date:** May 2025 (closed)

> "BrowserType.connect_over_cdp: Timeout 30000ms exceeded. Call log: <ws connect> ws://localhost:1234/devtools/browser/... - <ws connected> ws://localhost:1234/devtools/browser/..."
> "A Chrome/Edge browser update (version 136.0.7103.93+) changed how remote debug ports work."

- [FACT] websocket connects successfully but times out at protocol init (v1.52.0)
- [FACT] chrome m136+ requires `--user-data-dir` parameter when launched with remote debug
- [KHUE] whether this is playwright issue or chrome behavioral change remains debated

#### [3] github issue #4054 — regression: chromium.connect with vanilla CDP
**url:** https://github.com/microsoft/playwright/issues/4054
**type:** [official]
**date:** 2020 (playwright v1.4.0, closed)

> "The core protocol has indeed changed in 1.4, and we are aware that this usecase is not supported anymore."
> "it was never a goal for Playwright to work against CDP-based remote websockets."

- [FACT] playwright v1.4.0 introduced proprietary wire protocol that replaced standard CDP
- [FACT] custom protocol provides 10x improvement in network roundtrips
- [OPIN] playwright team: CDP remote websocket support was "never a goal"
- [SUMP] services like browserless implement custom `/playwright` routes as workaround

#### [4] github issue #24252 — CDP connection issue (google chrome)
**url:** https://github.com/microsoft/playwright/issues/24252
**type:** [official]
**date:** July 2023 (closed)

> "Chrome runs for 12+ hours" before connection issues appear
> "Direct WebSocket communication and raw Node.js libraries work sporadically (4 successful connections out of 20 attempts with Puppeteer)"

- [FACT] long-run chrome instances (12+ hours) exhibit CDP connection degradation
- [FACT] `/json/version` endpoint remains responsive even when CDP connections fail
- [KHUE] root cause (chrome or playwright) remains unresolved
- [SUMP] official response redirected to chromium bug tracker (crbug.com)

#### [5] github issue #23566 — feature: log proactive websocket closure
**url:** https://github.com/microsoft/playwright/issues/23566
**type:** [official]
**date:** June 2023 (completed, PR #23689 merged)

> "this code isn't exactly friendly, because it doesn't perform any log as for: the fact that the websocket was proactively terminated by Playwright [and] the problematic non-JSON-parseable string that triggered the decision."

- [FACT] playwright proactively closes websocket connections when JSON parse fails
- [FACT] prior to PR #23689, closures were silent — caused "Error: Browser closed" messages
- [FACT] log was added in june 2023 to diagnose connection failures

#### [6] browser-use blog — leave playwright for CDP
**url:** https://browser-use.com/posts/playwright-to-cdp
**type:** [practitioner]
**date:** 2024-2025

> "When a tab crashes in the browser or some operation is performed without focus on a page correctly, there are edge cases where the node.js process can hang indefinitely while it waits for a browser reply."
> "Currently we have no recourse but to kill -9 and attempt to reconnect to the browser from scratch."
> "Unfortunately the double RPC through the node.js relay means some state inevitably drifts across the 3 places"
> "there are at least 10 different ways a tab can crash in Chrome"

- [FACT] three-point state divergence (browser, relay server, client) causes reliability issues
- [FACT] tab crashes can leave processes in unrecoverable suspend states
- [OPIN] direct CDP access more reliable than playwright abstraction layer
- [SUMP] event-driven monitor via native CDP events more reliable than poll

#### [7] github issue #23427 (openclaw) — CDP connection silently dies after idle
**url:** https://github.com/openclaw/openclaw/issues/23427
**type:** [practitioner]
**date:** 2025

> "After approximately 5-6 hours of browser idle time, the Playwright-to-Chrome CDP WebSocket connection silently dies"
> Gateway "reports cdpReady: true despite dead connection"
> "CDP WebSocket keepalive — periodic lightweight ping on the Playwright CDP channel (e.g., Runtime.evaluate({expression: '1'}) every few minutes)"

- [FACT] CDP connections can silently die after 5-6 hours idle
- [FACT] HTTP health checks pass while websocket channels are dead
- [SUMP] macos app nap and chrome internal cleanup may cause idle connection decay
- [SUMP] keepalive pings (`Runtime.evaluate`) every few minutes can maintain connection health

#### [8] cloudflare browser render documentation
**url:** https://developers.cloudflare.com/browser-rendering/cdp/playwright/
**type:** [official]
**date:** current

> `keep_alive=600000` parameter in websocket endpoint URL

- [FACT] `keep_alive` parameter (milliseconds) controls session duration in cloud CDP endpoints
- [SUMP] session management requires explicit timeout configuration for stability

### convergence

multiple sources agree:
- timeout issues are pervasive — websocket connects but protocol init times out
- long-run instances degrade — CDP stability degrades after hours of uptime
- health checks insufficient — HTTP endpoints report healthy while websocket dead
- keepalive is common solution — periodic pings maintain connection

### conflict

**root cause attribution:**
- playwright team (#24252): "file with chromium team at crbug.com"
- practitioners (browser-use): "three-point state drift" in playwright architecture
- [KHUE] definitive root cause remains disputed

**CDP vs playwright protocol:**
- official docs: "significantly lower fidelity" — use `browserType.connect()` instead
- practitioners: direct CDP provides "raw access" and eliminates relay issues
- [OPIN] trade-off between convenience and reliability is context-dependent

**reconnection capability:**
- playwright: no built-in automatic reconnection
- third-party services (browserless): support session reconnection via custom CDP commands
- [FACT] native playwright requires manual reconnection implementation

### known bugs and stability issues

| issue | versions affected | status | workaround |
|-------|------------------|--------|------------|
| websocket connects but times out | v1.50.0+ with chrome m136+ | requires `--user-data-dir` flag | add user-data-dir parameter |
| silent websocket closure on JSON parse errors | pre-june 2023 | fixed (PR #23689) | update playwright |
| CDP connection decay after 5-6 hours idle | all versions | open | implement keepalive pings |
| long-run chrome (12+ hours) connection failures | all versions | unresolved (redirected to chrome) | restart chrome periodically |

### recommended reconnection patterns

1. **keepalive implementation:** `Runtime.evaluate({expression: "1"})` every 2-5 minutes
2. **health check enhancement:** verify websocket command channel, not just HTTP endpoint
3. **auto-recovery:** detect timeouts and automatically restart browser process
4. **event-driven monitor:** subscribe to native CDP events for disconnect detection
5. **session timeout configuration:** use explicit `keep_alive` parameters where supported

### verdict

- [FACT] `connectOverCDP` is "significantly lower fidelity" than playwright protocol
- [FACT] chrome m136+ requires `--user-data-dir` for remote debug
- [FACT] idle connections silently die after 5-6 hours
- [FACT] long-run chrome (12+ hours) exhibits connection degradation
- [FACT] no built-in automatic reconnection in playwright
- [SUMP] keepalive pings (`Runtime.evaluate`) every 2-5 minutes recommended
- [SUMP] periodic browser restart (every 6-12 hours) prevents degradation
- [KHUE] root cause of degradation (chrome vs playwright) remains disputed

## synthesis

### implications for bhrowser port

| topic | risk | mitigation |
|-------|------|------------|
| playwright-extra compatibility | moderate — no updates since 2023 | pin to known-work playwright version, test before upgrade |
| StealthPlugin maintenance | low to moderate — functional for basic detection | document 3-tier escalation, expect 60-80% success |
| CDP reconnection stability | moderate — silent failures, idle decay | implement keepalive pings, periodic browser restart, robust error handle |

### recommended approach

1. **stealth:** accept 60-80% effectiveness limit; document escalation to headful + human for enterprise anti-bot
2. **CDP:** implement keepalive pings; wrap `connectOverCDP` with retry logic; restart browser every 6-12 hours
3. **playwright-extra:** pin version; monitor github issues; be prepared to fork or switch to python if node.js stagnates further
